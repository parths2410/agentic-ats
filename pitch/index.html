§ 1 · TL;DR

What we built, and why it's worth slotting into Spine.#

Agentic ATS is a working hiring assistant that ingests a job description, proposes scoring criteria, parses any number of resume PDFs into structured profiles, scores each candidate against those criteria with calibrated rationale, and then lets a recruiter have a conversation with the resulting candidate pool — “show me the senior backend engineers from fintech”, “sort by Python depth”, “why is candidate X ranked above Y?” — driving real UI mutations through LLM tool calls, not regex.

The interesting engineering is not the chat. It's the five subsystems that make the chat trustworthy and embeddable:

  1. An agentic loop that lives in the service layer (not inside the provider), runs up to five LLM turns, executes tool calls between them, and folds all UI side-effects from a turn into one atomic mutation payload.
  2. A tool contract (definitions / read tools / action tools / registry) that is the only surface the LLM uses to read or change application state — making the system auditable and easy to extend with HRMS-specific tools (schedule_interview, advance_to_round_2, send_offer_letter).
  3. A prompt strategy with four hand-written, schema-anchored prompts (criteria extraction, resume parsing, candidate scoring, and a tool-aware chat system prompt) versioned as Python modules in the repo, not in a vendor console.
  4. A resume pipeline with pdfplumber → OCR fallback → LLM structuring → LLM scoring, plus a stale_scores invariant that guarantees the rank you see is never from a previous criteria set.
  5. An LLM provider abstraction — a four-method ABC — so the entire system can be repointed at Azure OpenAI or Bedrock by writing one new file. (Important for a Microsoft-stack acquirer.)

§ 2 · System overview

One picture, then we go deep.#

The system is a stateless React SPA over a Python/FastAPI backend. The backend owns three things that matter: the chat service (where the agentic loop lives), the resume pipeline (where PDFs become scored candidates), and the LLM provider (a thin, swappable adapter). All three converge on a single SQLite database in v1; the schema is Postgres-ready and the migration path is well understood.

The blue boxes are the value: ChatService, the tool registry, and the provider ABC. The dashed boxes are the parts of the abstraction we have already designed for but not yet implemented (Azure OpenAI, Bedrock). Everything below this section is about earning each of those boxes.

§ 3 · The agentic loop

Why the loop lives in the service, not the provider.#

The most common mistake we see in agentic codebases is letting the LLM SDK own the loop. The Anthropic SDK has a tools parameter; you pass it tools and it calls them for you in some helpers. Convenient — and the moment you swap providers, your whole orchestration goes with it. Worse: you can no longer interpose between turns to stream tool_status events, accumulate UI mutations, or enforce a hard iteration cap.

Our split is precise: LLMProvider.chat() handles a single LLM turn — messages in, text-or-tool-calls out. The loop itself lives in ChatService.handle_message. Provider swaps don't touch loop logic. Tool dispatch, mutation accumulation, iteration cap, and event streaming all live in one place that is fully unit-tested.

The loop in code

Below is the actual handle_message method. Two things to notice: first, the loop body is small — most of the complexity is in the helpers it delegates to. Second, there is exactly one place where iteration is capped, and one place where mutations are merged. There are no hidden control paths.

Design note · why MAX_ITERATIONS = 5 Empirically, every legitimate user query in this domain — even compound ones like “sort by Python depth, then highlight the top 5 with fintech experience” — resolves in ≤ 3 iterations. Five gives headroom for one self-correction without ever letting the model thrash. When the cap is hit we surface truncated: true to the client; the UI can then offer the user a follow-up button.

Tool dispatch, with first-class observability

The _execute_calls helper is what fans the LLM's tool calls out to our registry. Two design points:

  1. Tool errors do not raise — they are returned to the LLM as a structured result. The model can then read the error, decide to try a different tool, or apologise to the user. This keeps the loop crash-free even with a buggy tool.
  2. Each call emits two tool_status WebSocket events (executing, then complete with a one-line summary). The reference UI uses these to render a live “the assistant is searching candidates by skills…” ticker. Spine HRMS could surface the same.
Honesty note Tool calls are dispatched serially today, not in parallel. The LLM rarely emits more than 2 tool calls per turn, and parallel SQLAlchemy writes would race on commit, so the savings aren't worth the session-management cost yet. The contract supports parallel dispatch — we have the asyncio scaffolding — and switching is a one-method change behind a per-tool “mutates state?” flag we already track.

The mutation accumulator: many tool calls, one UI update

Here is a subtlety that turns out to matter. Within a single chat turn the LLM might call remove_highlights([A, B]), then later call set_highlights([C, D, E]). Naïvely, the UI would receive two messages and flicker. Worse, the semantics of the second call (“replace the set”) would override the first (“remove from the set”) — so the right thing for the UI to do is just apply the final set. Generalising this for every pair of action tools is the job of the UIMutationsAccumulator.

The accumulator collapses any number of action-tool mutations from a single turn into one payload. Highlight semantics are subtle:

  • set_highlights replaces the highlight set. (See commit 847099d — older docs incorrectly described this as additive; the tool now does exactly what its name says.)
  • remove_highlights before any set in the turn is recorded as a delta against whatever the frontend currently shows.
  • remove_highlights after a set in the turn is folded directly into the target set.
  • clear_highlights empties everything; set_sort and reset_ui are last-write-wins.

Worked example: filter chaining

User says: “highlight the senior engineers, then keep only the ones with fintech experience.” The LLM reasonably emits: 

turn N — tool calls:
  1. set_highlights(ids=[a, b, c, d, e])              # five seniors
  2. get_ui_state()                                   # read for chaining
  3. search_candidates(field=skills, query="fintech") # finds [b, c, x, y]
  4. set_highlights(ids=[b, c])                       # intersection

accumulator state after turn:
  target_set = [b, c]
  → final ui_mutations = { "highlights": { "set": ["b", "c"] } }

# the UI receives ONE replace, not four mutations.

The accumulator is what makes the agentic loop ergonomic for an HRMS UI: complex multi-step reasoning collapses into one atomic state delta the UI can render in a single frame.

§ 4 · The tool contract

The only surface the LLM uses to read or change application state.#

A tool, in our world, is a JSON-schema-described function the LLM is allowed to call. Adding a new HRMS-specific capability — “schedule an interview slot”, “send the offer letter PDF”, “move candidate to round 2” — is a matter of writing one schema, one Python function, and one registry entry. Three files, no loop changes, no prompt changes.

Today's tool taxonomy

The 12 tools split into two categories: read tools query state and never mutate; action tools mutate UI state and return a mutation object that the accumulator folds into the final payload.

ToolKindPurpose
get_candidates readList candidates with rank, score, and basic profile.
get_candidate_detail readFull structured profile for one candidate.
get_candidate_raw_text readOriginal resume text — for nuance the structured profile loses.
get_candidate_scores readPer-criterion scores + rationale for one candidate.
search_candidates readSubstring search over a chosen profile field.
compute_stats readAggregate stats over the candidate pool.
get_ui_state readCurrent highlights + sort — the LLM reads this for filter chaining.
set_highlights actionReplace the highlight set with exactly these IDs.
remove_highlights actionRemove specific IDs from the current highlight set.
clear_highlights actionEmpty the highlight set.
set_sort actionRe-sort by aggregate or any criterion name.
reset_ui actionDrop highlights + sort back to defaults.

The four-file split, and why it pays off

Tools live under backend/app/tools/, deliberately fragmented by concern:

  • definitions.py — the JSON schemas exposed to the LLM, plus a ACTION_TOOL_NAMES set used by the loop to detect mutations.
  • data_tools.py — read-only executors. Pure functions of (db, role_id, args). Easy to test, easy to reason about, no commits.
  • action_tools.py — mutation executors. Same signature; each returns a mutation object alongside its data result.
  • registry.py — name → executor dispatch.

This split is why a Spine engineer can add a new HR tool in an afternoon: write the schema, write the executor with the same shape as everything else in the file, register it. The LLM picks it up automatically on the next chat turn — system prompts already advertise self.registry.definitions().

§ 5 · Prompt strategy

Four prompts, all in-repo, all schema-anchored.#

Prompts are code. We treat them like code: they live as Python modules under backend/app/llm/prompts/, they're versioned with the rest of the codebase, and they're imported by the services that need them. Nothing lives in a vendor console or a database table where it can drift.

The four prompts:

  1. extract_criteria.py — turns a JD into 5–10 scorable criteria.
  2. parse_resume.py — turns raw resume text into a structured profile.
  3. score_candidate.py — scores one candidate against all criteria in a single call.
  4. chat_system.py — the agentic chat system prompt. Tool-aware. Behavior-rule-driven.

The same patterns recur across all four:

  • Schema-anchored output. Each prompt declares the exact JSON schema it wants back, and explicitly demands “no prose, no code fences”. Downstream parsing is a single json.loads.
  • Calibration anchors. The scoring prompt has explicit anchor descriptions for each band of the 1–10 scale. This is what stops scores from clustering at 7.
  • Anti-confabulation rules. “Do not invent. If absent, set null. If ambiguous, lower the confidence score for that section.”
  • Behavior rules separated from context. The chat prompt has a fixed _BEHAVIOR_RULES block stitched onto a per-role context block — so we can iterate on rules without rebuilding context.

1. extract_criteria — JD → scorable criteria

2. parse_resume — raw text → structured profile

Two ideas worth highlighting. First, the schema declares confidence_scores per section — the LLM tells us how clean the source was, which we surface to the user when a profile looks sparse. Second, the rules are explicit about what counts as a skill: “specific (PostgreSQL, Kafka, React), not generic (programming, communication).” That one line removed a class of garbage from our profiles.

3. score_candidate — calibrated scoring with rationale

This is the most carefully engineered prompt in the system. The 1–10 scale has explicit anchor descriptions for every band, and the rules forbid score-padding by demanding cited evidence per criterion. Without these anchors, every candidate ends up at 7.

Why one call per candidate, not one per criterion The model sees all criteria at once, so a strong signal in one (“ten years of Python at a fintech”) properly informs adjacent ones (“backend depth”, “domain experience”) without us having to coordinate independent calls. We pay one round-trip per candidate, and the rationales hang together.

4. chat_system — the tool-aware system prompt

The chat system prompt has two parts: a per-role context block that's stitched at runtime (role title, JD, criteria with weights, candidate count), and a fixed _BEHAVIOR_RULES block that codifies how the model should use the tools it has. The rules are the result of bug-driven iteration — every line is there because the model used to do something else.

§ 6 · Resume → score pipeline

From PDF bytes to a ranked candidate, in four stages.#

Every uploaded resume runs through the same four-stage pipeline: text extraction, LLM structuring, LLM scoring, and rank computation. The pipeline is async and per-candidate; a 50-resume upload progresses concurrently and emits per-candidate progress events over a dedicated WebSocket.

Stage 1: text extraction with OCR fallback

Real-world resumes are messy. About 70% are text-native PDFs that pdfplumber handles trivially. Another ~20% are scans, photographed pages, or PDFs exported from image editors — text-native extraction returns 0–80 characters of garbage. The remaining slice are mixed (cover page is text, body is a scan). We handle all three:

The 80-character threshold (_MIN_TEXT_THRESHOLD) is the empirical “is this actually a resume” line; below it we'd be parsing page numbers and headers. The fallback compares OCR text length against the original — we only switch if OCR did better, so a partially-extractable PDF doesn't get downgraded to OCR's noisier output.

Stage 2 & 3: LLM structuring then scoring

Both stages call the LLMProvider ABC, never a vendor SDK directly. Stage 2 produces the JSON structured profile we showed in § 5; stage 3 takes that profile, the JD, and the criteria, and returns one score + rationale per criterion in a single call.

Errors are first-class. If structuring fails, the candidate gets status="error" with an error_message string; the user sees it in the UI and can retry. If scoring fails after structuring succeeded, we keep the structured profile and mark scores as stale, so a criteria edit later will retry naturally.

Stage 4: aggregate, rank, and the stale_scores invariant

Once all per-criterion scores are persisted, we compute a weight-multiplied aggregate and sort candidates by it to produce ranks. stale_scores is a per-candidate boolean we use to guarantee one thing: the rank you see on screen always reflects the current set of criteria. The flag is set true whenever criteria change; _persist_scores is the only place it's set back to false. 

# Lifecycle of stale_scores

  candidate uploaded         status=pending,   stale_scores=false
  scoring complete           status=complete,  stale_scores=false
  user adds/edits criterion  ─────────────────► stale_scores=true (all candidates)
  re-score triggered         scoring runs again
  re-score complete          stale_scores=false (cleared in _persist_scores)

The UI is free to show a “stale” badge while the re-score is in flight. The backend guarantees the boolean is correct.

Concurrency and the progress channel

A batch upload of N resumes processes them concurrently with bounded parallelism. Per-candidate progress events flow over the second WebSocket endpoint (/ws/roles/{role_id}/progress) so the UI can show candidate A is parsing, candidate B is scoring, candidate C errored in real time:

§ 7 · LLM provider abstraction

Anthropic today, Azure OpenAI tomorrow, no rewrites.#

The LLM is behind a four-method ABC. Every call site in the system — ChatService, CriteriaService, ResumeService — depends on the abstract interface. Repointing the whole system at Azure OpenAI is one new file plus a one-line constructor change.

Three things to notice. First, chat()'s docstring explicitly forbids provider-side looping — that's how we keep the loop logic in our code. Second, the inputs to chat() are normalised types (LLMMessage, LLMResponse), so nothing leaks vendor wire-format into the rest of the codebase. Third, the other three methods describe semantically what they return — not how — so a provider can use whichever native primitive it wants (function-calling, JSON mode, structured output) as long as the post-condition holds.

The Anthropic implementation, in 15 lines

The provider does four small jobs:

  1. Translate normalised LLMMessage objects to Anthropic wire format (and back, for the response).
  2. Retry transient errors (RateLimitError, APIConnectionError, InternalServerError) with exponential backoff.
  3. Parse the raw response into a LLMResponse with text and tool_calls.
  4. Stay out of the loop. Critically, no orchestration. One turn in, one turn out.

What an Azure OpenAI provider looks like

Below is the skeleton of the new file. The shape is identical to the Anthropic provider; the work is in the format-translators. We have already prototyped the translator pair against OpenAI's tool-use schema (it's structurally the same: tool_use blocks become tool_calls arrays, results become role-tool messages with a tool_call_id).

The four prompts (§ 5) work as-is. They are model-agnostic by design — schema-anchored output and explicit anti-confabulation rules don't change between providers. We have run all four prompts against GPT-4-class models in spike testing with no edits required.

For Spine — the Microsoft-stack story Azure OpenAI is almost certainly where Spine's enterprise customers want their data to flow. The provider abstraction means “deploy on Azure” is a one-file integration, not a product rewrite. Your customers get tenant-scoped Azure OpenAI deployments; we deliver the provider. Anthropic stays available for customers who prefer it.

§ 8 · Real-time protocol

Two WebSocket endpoints, well-defined messages.#

The system exposes exactly two WebSocket endpoints, both per-role. Everything else is REST.

EndpointDirectionFrames
/ws/roles/{id}/chat ↔︎ client → server: {type: "chat_message", content}
server → client: ready, tool_status (executing / complete + summary), chat_complete (text + ui_mutations + iterations + truncated), error
/ws/roles/{id}/progress ← server batch_started, progress (per-candidate, per-stage), batch_complete

The progress channel is fed by an in-process pub/sub (app/pipeline/progress.py) so the resume pipeline can publish events without knowing or caring which clients are connected. A reconnecting client doesn't lose events because the server replays the candidate state on connect — the source of truth is the database.

Coverage isn't the goal — confidence is. The places that approach 100% are the places where a bug would propagate widely (loop logic, tool dispatch, mutation accumulation, provider translation). The places below 100% are intentional: a few error branches in data_tools that are hard to fake without hostile inputs, and a couple of UI rendering paths that we asserted as “cosmetic, will drift”.

Tooling: pytest with asyncio_mode = auto (so async tests don't need a marker), pytest-cov for line coverage, vitest + jsdom on the frontend. There are also milestone smoke scripts under frontend/scripts/verify-m*.mjs — out-of-band sanity checks we run before tagging a milestone complete.
